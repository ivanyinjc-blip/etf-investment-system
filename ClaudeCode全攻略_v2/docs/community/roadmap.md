## 项目目标

**帮助中文用户从入门到团队落地**

## 开放共建方式

### 1. Pull Request（代码/文档贡献）

```
贡献流程：
1. Fork 本仓库
2. 创建分支：git checkout -b feature/你的功能名
3. 编写内容
4. 提交：git commit -m "feat: 添加 xxx 说明"
5. Push：git push origin feature/你的功能名
6. 提 PR：在 GitHub 页面创建 Pull Request
7. 等待 Review：一般 48 小时内回复
```

**贡献内容类型：**

| 类型 | 说明 | 存放目录 |
|------|------|---------|
| 文档修正 | 错别字、表达不清、链接失效 | `docs/` |
| 新增章节 | 有价值但缺失的内容 | `docs/` |
| 实战案例 | 你的真实使用经验 | `docs/cases/` |
| 配置模板 | 可复用的 config 示例 | `configs/` |
| Skills 模板 | 可复用的 Skill 工作流 | `skills/` |

### 2. Issue（问题反馈和建议）

| Issue 类型 | 模板 | 响应时间 |
|-----------|------|---------|
| Bug 报告 | 使用 Issue 模板 | 48 小时内 |
| 功能建议 | 使用 Discussion | 72 小时内 |
| 文档纠错 | 简短说明即可 | 72 小时内 |
| 问题求助 | 先搜文档，搜不到再提 | 不保证 |

**提 Issue 注意事项：**

```
✅ 这样做：
- 标题简洁："[文档] CLI 命令参考缺少 --verbose 说明"
- 描述清晰：说明问题和建议的修改
- 提供截图：让维护者一眼看懂

❌ 别这样做：
- "文档有问题"（没说哪个文档、什么问题）
- "能不能加个xxx"（没有说明为什么需要）
- 重复提问（先搜搜有没有类似的）
```

### 3. 翻译（将英文文档中文化）

**翻译流程：**

```
1. 在 Issue 中说明想翻译哪个文档
2. 维护者 assign 给你
3. 创建分支：git checkout -b translate/文档名-zh
4. 完成翻译，对照英文原版确保准确
5. 提 PR，标注 "翻译" + 原文链接
```

**翻译规范：**

| 规范 | 说明 |
|------|------|
| 术语统一 | 对照 [术语表](./terminology.md) 翻译 |
| 保留原文 | 首次出现的术语加括号标注英文 |
| 格式一致 | 标题层级、代码块、表格格式统一 |
| 校对一遍 | 提交前通读一遍，确保通顺 |

## 贡献者须知

### 格式规范

```markdown
# 标题用 H1（页面大标题）
## 二级标题用 H2
### 三级标题用 H3

代码块必须标注语言：
```bash
claude --help
\`\`\`

表格要对齐：
| 列1 | 列2 | 列3 |
|-----|-----|-----|
| 内容 | 内容 | 内容 |
```

### 内容标准

```
✅ 好的内容：
- 有实际代码示例
- 有步骤说明（1、2、3...）
- 说明了适用场景和限制
- 标注了资料来源

❌ 不好的内容：
- 只有理论没有实践
- 命令没有示例输出
- 没有说明哪个版本适用
- 大段引用没有标注来源
```

### 审核流程

```
提交 PR
    ↓
自动检查（格式、链接、错别字）
    ↓
维护者 Review（1-2 天）
    ↓
通过 → 合并到 main 分支 → 自动部署
拒绝 → 反馈修改意见
```

**被拒绝的常见原因：**

| 原因 | 说明 |
|------|------|
| 格式不符合 | 标题层级、代码块标注等不符合规范 |
| 内容重复 | 已有类似内容，重复建设 |
| 事实错误 | 技术细节有误，未经核实 |
| 质量不足 | 太简略，没有提供足够价值 |
| 不符合定位 | 内容与本项目定位不符 |
