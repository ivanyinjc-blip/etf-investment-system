# 06 - 任务执行机制

> 本章聚焦 Claude Code 的核心工作方式：Agent 循环、工具调用规则、小步修改原则，以及如何处理中断与恢复。

---

## 1. Agent 循环

Claude Code 的核心是一个持续运转的 **Agent 循环**：

```
用户提示 → 理解任务 → 规划步骤 → 调用工具 → 检查结果 → 响应 → 循环
```

每一次循环中，Claude 会：

1. **解析任务**：结合 Context、CLAUDE.md、项目结构理解要做什么
2. **制定计划**：决定调用哪个工具、操作哪些文件
3. **执行工具调用**：按顺序调用 `Read` → `Edit` → `Bash` 等工具
4. **验证结果**：检查工具输出是否符合预期
5. **决定下一步**：继续循环或向用户返回结果

> **理解这个循环，是掌握 Claude Code 的关键。** 你不是在下达命令，而是在引导一个自主工作的 Agent。

---

## 2. Claude 如何理解任务

Claude 对任务的理解来自多层叠加的上下文：

| 层次 | 来源 | 说明 |
|------|------|------|
| **用户提示** | 你输入的指令 | 最直接的任务描述 |
| **CLAUDE.md** | 项目根目录 | 项目级持久化规则和工作流 |
| **.claude/rules/** | 规则目录 | 路径范围特定的指令 |
| **项目结构** | 读取的代码文件 | 自动理解代码组织方式 |
| **工具输出** | 前面步骤的结果 | 动态更新的中间状态 |

### 提升理解质量的方法

- **给出具体的文件路径和行号**，减少 Claude 的搜索成本
- **描述期望结果**，而不是只说"改一下"
- **提供相关背景代码**，让 Claude 理解上下文
- **明确范围**，告诉 Claude 哪些文件不能碰

## 3. 工具调用规则

### 经典顺序：Read → Edit → Bash

```
1. Read（读取）→ 了解当前状态
2. Edit（修改）→ 小范围改动
3. Bash（验证）→ 运行测试/构建确认
```

这个顺序不是强制约束，而是最佳实践。核心原则是：

- **先读再写**：不读就改容易出低级错误
- **每改必验**：修改后立即运行测试或构建
- **顺序灵活**：根据实际情况调整

### 常见工具及其适用场景

| 工具 | 适用场景 | 典型用法 |
|------|---------|---------|
| `Read` | 了解当前代码状态 | 读取要修改的文件 |
| `Edit` | 单文件精确修改 | 改函数、调整逻辑 |
| `Write` | 创建新文件或全量覆盖 | 新建组件、生成配置文件 |
| `Bash` | 执行命令验证 | `npm test`、`git diff` |
| `Glob` | 批量查找文件 | 找所有测试文件、组件文件 |
| `Grep` | 搜索代码内容 | 找函数调用、检查引用 |
| `WebFetch` | 获取外部文档 | 查 API 文档、读参考文章 |

## 4. 小步修改原则

**每次只改一点，验证一点，再继续。** 这是 Claude Code 最重要的使用原则。

### 为什么小步修改更好？

| 优势 | 说明 |
|------|------|
| **降低风险** | 改动越小，引入 bug 的概率越低 |
| **快速反馈** | 每步都能验证，发现问题立即回退 |
| **易于调试** | 出问题时能精确定位是哪一步引入 |
| **减少 Token 消耗** | 上下文不会快速膨胀 |

### 实践技巧

- **不要一次性写完整模块**：先写框架，跑通后再填细节
- **每改一个函数就测试**：而不是改完所有函数再一起测
- **用 `git` 记录每步**：方便回退到稳定版本
- **遇到问题立即停**：不要带着未验证的改动继续前进

### 反面案例

> ❌ "帮我把这个项目从 JavaScript 改成 TypeScript"
> 
> 这是超大任务，容易中途失控。正确做法是拆解成：
> 1. 安装 TypeScript 依赖
> 2. 配置 tsconfig.json
> 3. 逐文件添加类型定义
> 4. 运行 tsc 修复错误

## 5. 遇到错误如何处理

当验证失败或遇到错误时，按以下优先级处理：

### 优先级 1：回退（最优先）

```bash
# 使用 Claude Code 的回退功能
/rewind        # 后退一步
/claude -r     # 从某个检查点恢复
```

或者手动回退 git：

```bash
git checkout -- .
git log --oneline -5  # 查看最近提交
git reset --hard <commit-hash>
```

### 优先级 2：简化

如果回退不可行，尝试简化问题：

- 注释掉最近添加的代码
- 恢复第三方依赖的默认配置
- 暂时跳过非核心功能

### 优先级 3：换方法

原路走不通时，换一条路：

- 换个 API 实现相同功能
- 用更简单的数据结构
- 分步骤绕过问题而不是一次性解决

### 常见错误处理模式

```
错误类型 → 检查顺序 → 修复方向
─────────────────────────────────
编译错误 → 语法检查 → 对照文档修正语法
测试失败 → 看断言 → 修正实现逻辑
运行时崩溃 → 看堆栈 → 修复异常处理
依赖冲突 → package.json → 升级/降级版本
```

## 6. 多文件修改的事务性保证

Claude Code **不提供** 原生事务性保证（即多文件要么全改成功、要么全回滚）。需要手动保证一致性。

### 实践策略

**策略一：Git 分支法**
```bash
git checkout -b feature/new-feature  # 创建分支
# ... 进行多文件修改 ...
git commit -m "feat: add new feature"  # 确认后提交
```

**策略二：修改清单法**

在开始前列出所有要改的文件，逐个确认：

```
要改的文件清单：
1. src/utils/helper.ts    ← 改什么：添加 formatDate 函数
2. src/components/UseCard.tsx  ← 改什么：引用新函数
3. tests/helper.test.ts    ← 改什么：添加测试用例
```

**策略三：验证节点法**

每改完一组相关文件，立即运行验证：

```
[阶段1] 改 utils/ → 运行 utils 测试 ✓
[阶段2] 改 components/ → 运行组件测试 ✓
[阶段3] 改 API/ → 运行集成测试 ✓
```

## 7. Session 恢复与状态管理

### /rewind 命令

`/rewind` 可以回退 Claude 的"思考历史"，让它从之前的某个状态重新开始：

```
/rewind 3 steps    # 后退 3 步
/rewind to read    # 回到读取文件那一步
```

适合场景：改错了方向，想回到某个检查点重来。

### claude -r（Resume Session）

如果 Claude Code 因故中断（如长时间无响应），可以使用：

```bash
# 在项目目录重新打开，会提示恢复上一个 session
claude

# 或者明确指定恢复
claude --resume
```

### 状态管理最佳实践

| 场景 | 推荐做法 |
|------|---------|
| 长时间任务 | 每完成一个阶段用 git commit 保存 |
| 多文件修改 | 写好修改清单，按清单顺序执行 |
| 中断风险 | 重要改动前先 `git add` 暂存 |
| 继续工作时 | 先 `Read` 关键文件了解当前状态 |

## 8. 任务中断与继续

### 主动中断

用户可以随时按 `Ctrl+C` 中断 Claude 的工作。Claude 会：

- 停止当前工具调用
- 保留已经修改的文件内容
- 等待用户的下一步指令

### 继续工作的正确姿势

**错误做法**：
> 直接输入"继续"（Claude 可能丢失上下文，不知道"继续"到哪里）

**正确做法**：
1. 先让 Claude 报告当前状态（"现在改到哪里了？"）
2. 明确告诉它下一步做什么（"继续改 userService.ts 的第 47 行"）
3. 或者提供检查点信息（"从刚才的测试失败那里继续修复"）

### /compact 命令：压缩上下文

当对话变长时，Claude 的上下文会膨胀。可以使用：

```
/compact    # 压缩历史，保留关键信息
```

这会让 Claude 总结之前的对话要点，然后在一个更精简的上下文中继续工作。
